Euryst: WhatsApp API limits

Il Problema

Euryst è un AI agent che vive su WhatsApp come layer cognitivo persistente per founder. Il problema dei limiti API di WhatsApp è critico perché:

  • Rate limits stringenti: WhatsApp Business API impone limiti sul numero di messaggi che possono essere inviati in un determinato periodo
  • Messaging tiers: Il sistema di tier (1K, 10K, 100K, unlimited) si sblocca gradualmente in base alla qualità delle conversazioni
  • Template message restrictions: I messaggi proattivi richiedono template pre-approvati
  • 24h session window: Dopo 24h dall'ultimo messaggio dell'utente, serve un template per ricontattarlo
  • Quality rating: Meta monitora la qualità (green/yellow/red) e può limitare o bannare l'account
⚠️ Rischio critico: Per un AI agent conversazionale che deve essere "sempre disponibile", questi limiti possono rompere l'esperienza utente e la value proposition di Euryst.

Approcci Comuni

1. Tier Management Strategy

Gestire attivamente la progressione attraverso i tier:

  • Iniziare con conversazioni di alta qualità per sbloccare tier superiori velocemente
  • Monitorare il quality rating costantemente
  • Evitare spam o messaggi non richiesti
  • Implementare feedback loops per migliorare le risposte

2. Session Window Optimization

✓ Soluzione tecnica:
  • Tracciare timestamp dell'ultimo messaggio utente
  • Inviare messaggi "keepalive" intelligenti prima dello scadere delle 24h
  • Usare template messages strategicamente per sessioni scadute
  • Preparare template library per diversi use case

3. Message Batching & Prioritization

Per un AI agent che potrebbe generare molte risposte:

  • Chunking intelligente: Aggregare informazioni in messaggi più densi invece di tanti piccoli messaggi
  • Priority queue: Messaggi critici vs informativi
  • Async processing: Non tutto deve essere real-time
  • User-initiated bias: Preferire risposte a domande dirette vs messaggi proattivi

4. Hybrid Architecture

✓ Architettura consigliata per Euryst:
  • WhatsApp come interfaccia principale per conversazioni real-time
  • Web dashboard complementare per contenuti long-form, analytics, configurazioni
  • Email digest per riepiloghi settimanali o notifiche non urgenti
  • Deep linking: Da WhatsApp al web per contenuti ricchi senza consumare messaggi

5. Template Strategy

Creare una libreria di template ben progettati:

  • Check-in templates: "Come sta andando il progetto X?"
  • Reminder templates: "Hai una deadline tra 2 giorni"
  • Insight templates: "Ho notato un pattern nei tuoi dati"
  • Re-engagement templates: Per sessioni scadute

Architettura Specifica per Euryst

Layer 1: Message Router

- Incoming message → immediate acknowledgment
- Queue processing con priority
- Rate limit aware scheduling

Layer 2: Cognitive Engine

- Context retrieval da memoria persistente
- Response generation (può essere async)
- Decide: send now, batch, defer, o redirect to web

Layer 3: Delivery Manager

- Traccia session windows
- Gestisce template fallback
- Monitora quality metrics
- Adaptive throttling

Prossimi Step per Ambra

Immediate (Questa settimana)

  1. Audit attuale setup: Che tier siamo? Qual è il quality rating? Quanti messaggi/giorno stiamo inviando?
  2. Implementare tracking: Dashboard minimale per monitorare limiti in real-time
  3. Session window management: Sistema per tracciare e gestire le 24h windows

Short-term (Prossime 2 settimane)

  1. Template library: Creare e far approvare 5-10 template essenziali
  2. Message prioritization: Implementare queue system con priorità
  3. Chunking logic: Aggregare risposte multiple in messaggi più densi
  4. Fallback strategy: Cosa succede quando hit limits? Email? Web notification?

Medium-term (Prossimo mese)

  1. Hybrid interface: Prototipo di web dashboard complementare
  2. Quality optimization: Sistema per massimizzare quality rating e sbloccare tier superiori
  3. Analytics pipeline: Capire pattern di utilizzo per ottimizzare message budget
  4. User education: Comunicare ai founder i limiti e come massimizzare il valore

Strategic Questions

  • Quanti messaggi al giorno servono VERAMENTE per essere utili a un founder?
  • Quali interazioni possono essere async vs real-time?
  • Come bilanciare "always available" con "respectful of attention"?
  • Il costo di WhatsApp Business API scala bene con il business model?

Considerazioni Finali

I limiti di WhatsApp API non sono un bug, sono una feature. Forzano a progettare interazioni di qualità. Per Euryst, questo significa:

  • Qualità > Quantità: Ogni messaggio deve portare valore
  • Context-aware: Usare la memoria persistente per evitare messaggi ridondanti
  • User-initiated bias: Lasciare che sia il founder a guidare la frequenza
  • Multi-channel thinking: WhatsApp è l'interfaccia, non l'unico touchpoint

Il vincolo diventa opportunità: costruire un AI agent che rispetta il tempo e l'attenzione del founder, non uno che spamma.
