Euryst: WhatsApp API Limits Deep Dive

📋 Context Refresh
Euryst è un AI agent che vive su WhatsApp come layer cognitivo persistente per founder. Il problema dei limiti API è critico perché può rompere l'esperienza conversazionale che è il core value del prodotto.

🔍 Il Problema in Dettaglio

WhatsApp Business API impone limiti stringenti che impattano direttamente su Euryst:

1. Tier System & Rate Limits

Tier Messaggi/24h Come Sbloccare Tempo Tipico
Tier 1 1,000 Automatico (nuovo account) Immediato
Tier 2 10,000 Quality rating + volume ~7 giorni
Tier 3 100,000 Quality rating + volume ~30 giorni
Unlimited No limit Quality rating + volume ~90 giorni
⚠️ Rischio Critico per Euryst
Con Tier 1 (1K msg/day), se Euryst ha 50 utenti attivi, sono solo 20 messaggi per utente al giorno. Per un "layer cognitivo persistente", questo è molto limitante.

2. Quality Rating System

Meta assegna un quality rating che influenza tutto:

  • GREEN: High quality - nessuna limitazione
  • YELLOW: Medium quality - possibili limitazioni
  • RED: Low quality - severe limits o ban

Fattori che influenzano il rating:

  • User blocks (utenti che bloccano il numero)
  • Reports (segnalazioni spam)
  • Message delivery failures
  • User engagement rate
  • Response time to user messages

3. Session Window (24h Rule)

User sends message → Session OPEN (24h) ├─ Durante le 24h: free-form messages OK └─ Dopo 24h: SOLO template messages (pre-approved) Template approval time: 1-48 hours Template rejection rate: ~30% (first submission)
⚠️ Problema per Euryst
Se un founder non scrive per 24h, Euryst non può fare check-in proattivi senza template. Ma i template sono rigidi e vanno contro l'idea di "conversazione naturale".

4. Template Message Constraints

  • Devono essere pre-approvati da Meta (1-48h review)
  • Formato rigido con placeholder limitati
  • Categorie limitate (marketing, utility, authentication)
  • No personalizzazione dinamica complessa
  • Rate limits separati per template

5. Throughput Limits

Oltre ai daily limits, ci sono anche:

  • 80 msg/second per phone number (burst)
  • Concurrent connections: limitate per tier
  • Media upload limits: 5MB per file, formati specifici

💡 Approcci Strategici per Euryst

Approccio 1: Smart Message Budgeting

✓ Implementazione Consigliata
1. Calcola budget giornaliero per utente daily_budget = (tier_limit * 0.8) / active_users 2. Prioritizza messaggi: - P0: Risposte a domande dirette (always send) - P1: Notifiche critiche (deadline, alert) - P2: Check-in proattivi (se budget disponibile) - P3: Insights/suggestions (best effort) 3. Batch & aggregate quando possibile: Instead of: "Task A done" + "Task B done" + "Task C done" Send: "✅ 3 tasks completed: A, B, C"

Approccio 2: Hybrid Interface Architecture

WhatsApp come "control center", non unico touchpoint

Euryst Architecture: ├─ WhatsApp (conversational interface) │ ├─ Quick interactions │ ├─ Time-sensitive alerts │ └─ Voice/text input │ ├─ Web Dashboard (deep work interface) │ ├─ Analytics & reports │ ├─ Long-form content │ ├─ Configuration │ └─ History search │ ├─ Email Digest (async updates) │ ├─ Weekly summaries │ ├─ Non-urgent insights │ └─ Deep-dive reports │ └─ Push Notifications (mobile app future) └─ Critical alerts only

Vantaggi per Euryst:

  • Riduce pressione su WhatsApp API limits
  • Offre interfacce ottimizzate per diversi use case
  • Mantiene WhatsApp per ciò che fa meglio: conversazione veloce
  • Permette contenuti ricchi (grafici, tabelle) sul web

Approccio 5: Multi-Number Architecture (Advanced)

Per scale maggiore:

  • Pool di numeri WhatsApp Business
  • Load balancing intelligente tra numeri
  • Ogni numero gestisce subset di utenti
  • Fallback automatico se un numero raggiunge limits
⚠️ Tradeoff
Più complesso da gestire, costi maggiori, ma necessario per scale 1000+ utenti attivi.

Core Components

Implementazione Message Prioritization

# Python example con Redis import redis from enum import Enum class Priority(Enum): P0_CRITICAL = 0 P1_HIGH = 1 P2_MEDIUM = 2 P3_LOW = 3 class EurystMessageQueue: def __init__(self): self.redis = redis.Redis() self.daily_limit = 1000 # Start with Tier 1 def can_send_message(self, user_id, priority): today = datetime.now().date() key = f"msg_count:{today}" current_count = self.redis.get(key) or 0 # Reserve 20% budget for P0 messages if priority == Priority.P0_CRITICAL: return current_count < self.daily_limit # P1-P3 share remaining 80% return current_count < (self.daily_limit * 0.8) def enqueue_message(self, user_id, content, priority): if not self.can_send_message(user_id, priority): if priority.value <= Priority.P1_HIGH.value: # Queue for next day self.redis.lpush(f"queue:tomorrow", { 'user_id': user_id, 'content': content, 'priority': priority.value }) return False # Send immediately self.send_whatsapp_message(user_id, content) self.redis.incr(f"msg_count:{datetime.now().date()}") return True

🚀 Conclusioni

I limiti di WhatsApp API sono reali e stringenti, ma non sono un blocco per Euryst. Sono un design constraint che forza qualità.

✓ Key Takeaways
  • Quality over Quantity: Ogni messaggio deve portare valore reale
  • Hybrid by Design: WhatsApp è il control center, non l'unico touchpoint
  • Smart Prioritization: Non tutti i messaggi sono uguali
  • Template Strategy: Preparare template versatili e conversazionali
  • Monitor & Adapt: Quality rating e tier progression sono critical metrics

Il vincolo diventa opportunità: costruire un AI agent che rispetta il tempo e l'attenzione del founder, non uno che compete per attention con notifiche spam.
