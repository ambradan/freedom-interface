Euryst: memory persistence

Il Problema

Euryst è un AI agent WhatsApp che funge da "layer cognitivo persistente" per founder. Il challenge principale è mantenere memoria coerente e contestuale attraverso conversazioni WhatsApp distribuite nel tempo.

⚠️ Challenge Core: WhatsApp Business API è stateless. Ogni messaggio è un evento isolato. Serve architettura di memoria che persista identità, contesto, decisioni e relazioni nel tempo.

Architetture di Memory Persistence

1. Hierarchical Memory System

Ispirato a sistemi cognitivi umani:

  • Working Memory: ultimi N messaggi in cache (Redis/Upstash)
  • Episodic Memory: conversazioni complete con embedding semantico (Qdrant/Pinecone)
  • Semantic Memory: fatti estratti, entità, relazioni (Neo4j/PostgreSQL)
  • Procedural Memory: pattern di interazione, preferenze utente (key-value store)
✓ Per Euryst: Founder hanno conversazioni ricorrenti su progetti, decisioni, metriche. Serve semantic layer che colleghi "quel problema di scaling di TenderMatch" menzionato 3 settimane fa con "i nuovi limiti database" di oggi.

2. Graph-Based Context

// Neo4j schema example
(Founder)-[:WORKS_ON]->(Project)
(Project)-[:HAS_CHALLENGE]->(Problem)
(Conversation)-[:MENTIONS]->(Problem)
(Decision)-[:ADDRESSES]->(Problem)
(Decision)-[:MADE_IN]->(Conversation)

Vantaggi:

  • Query contestuali: "Quali decisioni ho preso su questo problema?"
  • Evoluzione temporale: tracking di come pensiero cambia
  • Relazioni implicite: scoperta pattern cross-progetto

3. Vector Memory + Metadata Filtering

Ogni conversazione → embedding + metadata strutturati:

{
  "vector": [...],
  "metadata": {
    "user_id": "founder_123",
    "project": "TenderMatch",
    "topic": "scaling",
    "sentiment": "concerned",
    "entities": ["PostgreSQL", "Railway", "costs"],

  }
}

    ├→ Extract entities (NER)

    ├→ Recent messages (Redis)

    ├→ System prompt + founder profile
    ├→ Relevant past conversations
    └→ Current thread

    ↓
Memory Update Layer
    ├→ Store message + embedding
    ├→ Update graph relationships
    └→ Refresh working memory
    ↓
WhatsApp Business API

Persistent Storage Stack

  1. PostgreSQL/Supabase: conversazioni raw, metadata, user profiles
  2. Qdrant/Pinecone: vector embeddings per semantic search
  3. Neo4j/Memgraph: knowledge graph (opzionale ma potente)
  4. Redis/Upstash: working memory cache, session state

Challenges Specifici WhatsApp

Rate Limits:
  • WhatsApp Business API ha limiti su messaggi/secondo
  • Template messages vs session messages (24h window)
  • Serve queue system (BullMQ/Celery) per gestire backpressure

Session Management

WhatsApp ha "customer service window" di 24h dopo ultimo messaggio utente. Dopo, serve template pre-approvato.

Strategia:

  • Tracking ultimo messaggio utente per ogni conversazione
  • Proactive memory consolidation quando window sta per chiudersi
  • Template messages per "memory refresh" ("Ehi, 3 giorni fa parlavi di X, aggiornamenti?")

Memory Consolidation Strategies

1. Nightly Batch Processing

# Pseudocode
async def consolidate_memories():
    conversations = get_today_conversations()
    for conv in conversations:
        # Extract key decisions
        decisions = extract_decisions(conv)
        
        # Update knowledge graph
        update_graph(decisions)
        
        # Generate summary embedding
        summary = summarize_conversation(conv)
        store_embedding(summary)
        
        # Prune working memory
        cleanup_cache(conv.user_id)

2. Incremental Updates

Ogni messaggio triggera micro-updates:

  • Embedding immediato in vector store
  • Entity extraction e graph update
  • Cache refresh per working memory

Metriche di Successo

  • Context Recall: % volte che agent ricorda info rilevante passata
  • Response Coherence: coerenza con conversazioni precedenti
  • Latency: tempo retrieval + generation < 3s
  • Storage Efficiency: costo per utente/mese

Rischi & Mitigazioni

  • Cost explosion: vector storage scala con conversazioni → implement pruning strategy
  • Context pollution: troppa memoria = noise → semantic filtering + relevance scoring
  • WhatsApp bans: rispetta rate limits, avoid spam patterns
  • Privacy: founder data sensibili → encryption at rest, GDPR compliance
