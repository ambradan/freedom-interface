CDSS: UX medico

⚠️ Context Critico: Un CDSS per MDR Class IIa richiede UX che supporti il ragionamento clinico senza sostituirlo. L'interfaccia medico è il punto dove si decide se il sistema amplifica o ostacola il decision-making.

Il Problema

L'interfaccia medico in un CDSS deve bilanciare requisiti contrastanti:

Requisito Tensione
Densità informativa vs. Cognitive overload
Velocità accesso vs. Accuratezza verifica
Automazione vs. Controllo manuale
Standardizzazione vs. Personalizzazione workflow
Compliance MDR vs. Fluidità UX

Nel contesto CDSS specifico (Next.js 16, PostgreSQL 16, MinIO su Hetzner), il problema si manifesta in:

  • Gestione documenti multi-pagina (PDF, DICOM, immagini)
  • Visualizzazione evidenze con traceability completa
  • Workflow ipotesi diagnostiche ranked vs. non-ranked
  • Interfaccia paziente vs. interfaccia medico (role-based UI)
  • Performance su dataset clinici reali (latenza query, rendering)

Principi UX per CDSS Medicale

1. Cognitive Load Management

Regola d'oro: Un medico in pronto soccorso ha ~2 minuti per paziente. L'interfaccia deve surfacing le informazioni critiche in <5 secondi.

Pattern efficaci:

  • Progressive disclosure: Overview → Dettaglio → Evidence chain
  • Visual hierarchy: Colore per urgency (red flags), dimensione per confidence
  • Chunking: Max 7±2 elementi per schermata (Miller's Law)
  • Scannability: Bullet points, bold keywords, whitespace

2. Traceability & Trust

Per MDR Class IIa, ogni output deve essere tracciabile a fonte primaria.

UX Requirements:

  • Ogni dato clinico → link diretto a documento sorgente
  • Hover su diagnosi → snippet estratto + confidence score
  • Click → PDF viewer con highlight del testo estratto
  • Cronologia modifiche (chi, quando, perché)

3. Workflow-Centric Design

Non progettare per "utente generico medico". Progetta per task clinici specifici.

Task Priorità UX Metriche
Triage PS Red flags visibili in 3 sec Time to critical info
Visita ambulatorio Storia clinica completa Completeness score
Follow-up Delta vs. visita precedente Change detection accuracy
Refertazione Export strutturato Export format compliance

4. Error Prevention & Recovery

In ambito clinico, gli errori hanno conseguenze reali.

Pattern di sicurezza:

  • Confirmation dialogs: Per azioni irreversibili (delete documento, modifica diagnosi)
  • Undo stack: Cronologia ultime 10 azioni con rollback
  • Conflict resolution: Se AI suggerisce diagnosi contraddittoria con storico
  • Graceful degradation: Se OCR fallisce, mostra PDF raw + input manuale
Anti-pattern da evitare:
❌ Bottoni "Delete" rossi senza confirm
❌ Auto-save senza visual feedback
❌ Error messages generici ("Errore sconosciuto")
❌ Timeout senza retry automatico

Implementazione Stack-Specific

Next.js 16 + TypeScript

Vantaggi per UX medico:

  • Server Components: Rendering lato server per dati sensibili (no esposizione client)
  • Streaming SSR: Progressive loading per documenti pesanti
  • Route handlers: API endpoints type-safe per validazione input clinico
  • Middleware: Role-based access control (medico vs. paziente)
Pattern - Protected Route:
// middleware.ts
export function middleware(req: NextRequest) {
const role = req.cookies.get('role')?.value;
if (req.nextUrl.pathname.startsWith('/medico')) {
if (role !== 'CLINICIAN') return NextResponse.redirect('/login');
}
return NextResponse.next();
}

PostgreSQL 16

UX Performance:

  • Full-text search: Ricerca veloce su extracted_text documenti
  • JSON columns: Metadata flessibili senza schema rigido
  • Materialized views: Dashboard aggregati pre-calcolati
  • Row-level security: Paziente vede solo i suoi dati
Query Ottimizzata - Dashboard:
CREATE MATERIALIZED VIEW patient_summary AS
SELECT p.id, p.name,
COUNT(DISTINCT d.id) as doc_count,
COUNT(DISTINCT dx.id) as diagnosis_count,
MAX(v.date) as last_visit
FROM patients p
LEFT JOIN documents d ON d.patient_id = p.id
LEFT JOIN diagnoses dx ON dx.patient_id = p.id
LEFT JOIN visits v ON v.patient_id = p.id
GROUP BY p.id;

-- Refresh ogni ora
REFRESH MATERIALIZED VIEW CONCURRENTLY patient_summary;

Testing UX Medico

Metodologia

  1. Heuristic Evaluation (Nielsen's 10 usability heuristics)
  2. Cognitive Walkthrough (task-based, con medici reali)
  3. A/B Testing (varianti layout per task critici)
  4. Analytics (heatmaps, session recordings, funnel analysis)

Metriche Chiave

Metrica Target Tool
Time to First Interaction < 2 sec Lighthouse, Web Vitals
Task Success Rate > 95% User testing
Error Rate < 5% Sentry, logs
System Usability Scale (SUS) > 70 Questionario post-test
Net Promoter Score (NPS) > 50 Survey periodici

Compliance MDR & UX

L'interfaccia deve supportare requisiti regolatori:

Annex I - General Safety Requirements

  • Chapter II, 5.1: "Devices shall be designed to minimize risks arising from errors"
  • UX Impact: Error prevention, confirmation dialogs, undo

Annex I - Chapter III (Information)

  • 23.1: "Instructions for use shall be provided with each device"
  • UX Impact: Contextual help, tooltips, onboarding wizard

IEC 62366 (Usability Engineering)

  • Identificazione use scenarios e use errors
  • Validazione con summative usability testing (15+ utenti)
  • Documentazione usability engineering file
✅ Checklist MDR-UX:
□ Use scenarios documentati
□ Risk analysis per use errors
□ Formative testing (iterativo)
□ Summative testing (finale)
□ Instructions for use integrate in UI
□ Training materials per clinici

Next Steps per Ambra

Immediate (Settimana 1-2)

  1. Audit UX esistente: Registra 5 sessioni uso reale (se già in pilot)
  2. Identifica pain points: Quali task richiedono >3 click? Dove gli utenti si bloccano?
  3. Prioritize fixes: Matrice impatto/sforzo
  4. Quick wins: Implementa top 3 fix a basso sforzo

Short-term (Mese 1)

  1. Design system: Setup Radix UI + Tailwind, definisci componenti base
  2. Formative testing: 5 medici, think-aloud su task critici
  3. Iterate: 2-3 cicli design → test → refine
  4. Performance baseline: Lighthouse audit, Web Vitals monitoring

Mid-term (Mese 2-3)

  1. Summative testing: 15+ medici, task success rate, SUS score
  2. Analytics setup: Hotjar, Sentry, custom events
  3. Documentation: Usability engineering file per MDR
  4. Training materials: Video tutorials, quick reference guide

Long-term (Ongoing)

  1. A/B testing: Varianti layout per ottimizzazione continua
  2. User feedback loop: Survey trimestrali, NPS tracking
  3. Accessibility audit: WCAG 2.1 AA compliance
  4. Internationalization: Se target mercati esteri

Anti-Patterns da Evitare

🚫 Errori comuni CDSS UX:
  • "Feature creep": Aggiungere funzioni senza validare uso reale
  • "Designer-driven": UI bella ma non funzionale per workflow clinico
  • "Tech-first": Scegliere stack prima di capire requisiti UX
  • "One-size-fits-all": Stessa UI per PS, ambulatorio, domiciliare
  • "Alert fatigue": Troppi warning → medici li ignorano
  • "Black box AI": Suggerimenti senza spiegazione → no trust
  • "Mobile-last": Medici usano tablet, non solo desktop

Conclusioni

UX medico per CDSS è un problema di design sistemico, non solo estetico.

Principi guida:

  1. Workflow-centric: Progetta per task, non per utenti generici
  2. Trust through transparency: Ogni dato → fonte tracciabile
  3. Safety by design: Error prevention > error handling
  4. Performance is UX: Latenza >2 sec = frustrazione
  5. Iterate with users: 5 medici valgono più di 50 designer
💡 Raccomandazione finale: Inizia con 1 workflow critico (es. triage PS), ottimizzalo fino a SUS >80, poi espandi. Non cercare di fare tutto subito.
