🎯 BenchmarkPA: Data Sourcing Prezzi PA

📋 Contesto Progetto

Progetto: BenchmarkPA

Obiettivo: Sistema di pricing B2B2G per Pubbliche Amministrazioni con analisi sottosoglia

Stack: Supabase Edge Functions, PostgreSQL

Problema critico: Sourcing affidabile e strutturato dei dati prezzi delle gare PA

🔍 Analisi del Problema

Sfide Specifiche del Data Sourcing PA

  • Frammentazione delle fonti: Dati distribuiti su portali regionali, comunali, ANAC, CONSIP, MEF
  • Formati eterogenei: PDF, XML, CSV, HTML con strutture non standardizzate
  • Qualità variabile: Dati incompleti, errori di digitazione, categorizzazioni inconsistenti
  • Sottosoglia complicati: Gare sotto €40k spesso non pubblicate centralmente
  • Aggiornamento asincrono: Tempistiche di pubblicazione diverse per fonte
  • Normativa GDPR: Attenzione ai dati sensibili in documenti pubblici
⚠️ Criticità Sottosoglia: Le gare sotto soglia comunitaria sono il 70% del volume ma hanno la peggiore copertura dati. Rappresentano il vero valore differenziante di BenchmarkPA.

💡 Approcci Comuni per il Data Sourcing PA

1. API Istituzionali

Fonti primarie:

  • ANAC (Autorità Anticorruzione): dati.anticorruzione.it - Dataset Open Data XML/JSON
  • CONSIP: Convenzioni e prezzi di riferimento
  • MEF OpenData: Dati finanziari aggregati
  • Portali regionali: Lombardia, Emilia-Romagna hanno API strutturate

2. Web Scraping Strutturato

Per portali senza API (comuni, province piccole):

  • Puppeteer/Playwright: Automazione browser per portali dinamici
  • Cheerio: Parsing HTML statico
  • Scheduling: Cron jobs giornalieri/settimanali
  • Proxy rotation: Evitare blocchi IP
💡 Best Practice: Implementare rate limiting rispettoso (1-2 req/sec) e user-agent identificabile con contatto email per segnalazioni.

3. OCR e Document Processing

Per PDF e documenti scansionati:

  • Tesseract.js: OCR open source in JavaScript
  • Google Document AI: Parsing avanzato con ML
  • AWS Textract: Estrazione tabelle e form
  • Apache Tika: Metadata extraction multi-formato

4. Crowdsourcing e Segnalazioni

  • Form per PA che auto-caricano dati
  • Integrazione email forwarding (gare@benchmarkpa.it)
  • Bot Telegram per segnalazioni rapide

CREATE TABLE gare_pa (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  cig VARCHAR(10) UNIQUE,
  ente_appaltante TEXT,
  oggetto TEXT,
  importo_base DECIMAL(12,2),
  importo_aggiudicazione DECIMAL(12,2),

// Starter Edge Function (functions/ingest-anac/index.ts)
import { serve } from 'https://deno.land/std@0.168.0/http/server.ts';
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2';

serve(async (req) => {
  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  );

  // Fetch ANAC data
  const anacResponse = await fetch('URL_DATASET_ANAC');
  const data = await anacResponse.text();
  
  // Parse e insert
  const parsed = parseANACXML(data);
  const { error } = await supabase
    .from('gare_pa')
    .upsert(parsed, { onConflict: 'cig' });

  return new Response(JSON.stringify({ 
    inserted: parsed.length,
    error 
  }));
});

Metriche di Successo

  • Coverage: >10.000 gare indicizzate entro 30 giorni
  • Freshness: Dati aggiornati con latenza <24h
  • Quality: <5% righe con errori di parsing
  • Sottosoglia: >30% gare sotto €40k (differenziatore chiave)

⚡ Considerazioni Tecniche Supabase

  • Edge Functions timeout: Max 60s, usare queue per job lunghi
  • PostgreSQL limits: Free tier 500MB, pianificare retention policy
  • Rate limiting: Implementare con pg_cron per evitare ban IP
  • Costi: Pro plan ~$25/mese necessario per production load
