TenderMatch: scaling

Il Problema

TenderMatch è un compliance engine per gare pubbliche che analizza bandi tramite AI. Lo stack attuale (Lovable, Express, Railway, Supabase, PostgreSQL, Stripe, Make) funziona per MVP, ma presenta limiti critici di scalabilità:

Bottleneck identificati:
  • AI Processing: Analisi bandi CPU-intensive, rischio timeout su Railway
  • Database: PostgreSQL su Supabase può saturarsi con migliaia di bandi concorrenti
  • Make Automations: Ottimo per prototipare, ma costoso e lento su volumi alti
  • Monolith Express: Difficile scalare selettivamente le funzioni più cariche

Architettura di Scaling

1. Separazione Concerns (Priority #1)

Microservices pattern:

  • API Gateway (Lovable frontend) → rimane su Railway
  • Tender Ingestion Service → worker asincrono per scraping bandi
  • AI Analysis Service → queue-based processing (BullMQ + Redis)
  • Notification Service → gestione alert e match

Stack consigliato: Express → Supabase Edge Functions + BullMQ

2. Database Optimization

Supabase PostgreSQL tuning:

  • Partizionamento tabelle bandi per data pubblicazione
  • Indici BRIN per timestamp, GIN per full-text search
  • Read replicas per query analitiche (Supabase Pro)
  • Caching layer con Redis per bandi frequenti

Schema esempio:

CREATE INDEX CONCURRENTLY idx_bandi_published ON bandi USING BRIN(published_at);
CREATE INDEX idx_bandi_fulltext ON bandi USING GIN(to_tsvector('italian', description));

3. AI Processing Queue

Async processing con BullMQ:

  • Queue tender-analysis con priorità (urgenti vs batch)
  • Worker pool scalabile (Railway autoscaling o Render)
  • Rate limiting per API LLM (evitare 429)
  • Retry logic con exponential backoff

Implementazione:

// worker.js
const queue = new Queue('tender-analysis', { connection: redis });
queue.process(10, async (job) => {
  const analysis = await analyzeTenderWithAI(job.data.tenderId);
  await supabase.from('analyses').insert(analysis);
});

4. Sostituire Make.com

Make è costoso su scala (€0.10-0.20 per esecuzione). Alternative:

  • Supabase Edge Functions per webhook/notifiche (gratis fino a 500k invocazioni)
  • n8n self-hosted su Railway (open source, €0 oltre infra)
  • Temporal.io per workflow complessi (overkill per ora)

Migrazione graduale: Make → Edge Functions per notifiche semplici

Metriche di Scaling

Metrica Attuale (MVP) Target (Scale)
Bandi/giorno ~100 10.000+
Tempo analisi AI 30-60s <10s (async)
Utenti concorrenti 10-50 1000+
Costo/bando €0.50 <€0.05

Risorse Utili

  • BullMQ Documentation - Queue robuste per Node.js
  • Railway Autoscaling
  • n8n Self-Hosted - Alternativa a Make
  • PostgreSQL Partitioning
