TenderMatch: scaling

Il Problema

TenderMatch è un compliance engine per gare pubbliche che deve gestire:

  • Volume di bandi: migliaia di documenti al giorno da analizzare
  • Complessità AI: analisi semantica profonda di documenti tecnici/legali
  • Latenza critica: notifiche real-time per scadenze imminenti
  • Multi-tenancy: isolamento dati tra clienti enterprise
Collo di bottiglia principale: Lo stack attuale (Lovable + Express + Railway) non è progettato per workload asincroni pesanti. L'AI processing sincronizzato con le richieste HTTP crea timeout e degrado delle performance.

Architettura Attuale vs Scalabile

Stack Attuale (Problematico)

User Request → Express → AI Analysis → Response
                ↓
            Timeout (>30s)

Architettura Target

Ingestion Layer (Railway Workers)
    ↓
Message Queue (Redis/BullMQ)
    ↓
AI Processing Pool (Horizontal scaling)
    ↓
Supabase (Storage + Real-time notifications)
    ↓
Frontend (Lovable + WebSockets)

Soluzioni Concrete

1. Queue-Based Processing

Stack: BullMQ + Redis (Railway addon)

// worker.js
import { Queue, Worker } from 'bullmq';

const tenderQueue = new Queue('tender-analysis', {
  connection: process.env.REDIS_URL
});

const worker = new Worker('tender-analysis', async job => {
  const { tenderId, documentUrl } = job.data;
  
  // AI analysis asincrona
  const result = await analyzeWithAI(documentUrl);
  
  // Store in Supabase
  await supabase.from('tenders')
    .update({ analysis: result })
    .eq('id', tenderId);
    
  // Notifica real-time
  await supabase.channel('tender-updates')
    .send({ type: 'analysis_complete', tenderId });
});

2. Horizontal Scaling su Railway

Railway supporta replica scaling:

# railway.json
{
  "services": {
    "api": {
      "replicas": 3,
      "healthcheck": "/health"
    },
    "worker": {
      "replicas": 5,
      "command": "node worker.js"
    }
  }
}

Costo stimato: ~$50-100/mese per 5 worker instances

3. Database Optimization (Supabase)

-- Partitioning per tenant
CREATE TABLE tenders_partitioned (
  id UUID,
  tenant_id UUID,
  data JSONB,
  created_at TIMESTAMPTZ
) PARTITION BY LIST (tenant_id);

-- Index per query frequenti
CREATE INDEX idx_tenders_status 
ON tenders(status, created_at DESC) 
WHERE status IN ('pending', 'analyzing');

-- Materialized view per analytics
CREATE MATERIALIZED VIEW tender_stats AS
SELECT tenant_id, 
       COUNT(*) as total,
       AVG(processing_time) as avg_time
FROM tenders
GROUP BY tenant_id;

4. Caching Strategico

// Redis cache per risultati AI comuni
const getCachedAnalysis = async (documentHash) => {
  const cached = await redis.get(`analysis:${documentHash}`);
  if (cached) return JSON.parse(cached);
  
  const result = await runAIAnalysis(document);
  await redis.setex(`analysis:${documentHash}`, 3600, JSON.stringify(result));
  return result;
};

Migration Path

Fase 1: Quick Wins (1-2 settimane)

  1. Implementare BullMQ per AI processing
  2. Aggiungere Redis caching (Railway addon: $10/mese)
  3. Ottimizzare query Supabase esistenti
  4. Implementare rate limiting per proteggere l'infra

Fase 2: Scaling Infra (2-4 settimane)

  1. Separare worker processes da API server
  2. Configurare auto-scaling su Railway
  3. Implementare database partitioning
  4. Setup monitoring (Sentry + Railway metrics)

Fase 3: Advanced (1-2 mesi)

  1. Microservices per AI processing (se necessario)
  2. CDN per documenti statici (Cloudflare R2)
  3. Multi-region deployment per latency
