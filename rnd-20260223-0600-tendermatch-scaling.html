TenderMatch: scaling

Il Problema

TenderMatch è un compliance engine per gare pubbliche che analizza bandi usando AI. Lo stack attuale (Lovable, Express, Railway, Supabase, PostgreSQL, Stripe, Make) funziona per MVP, ma presenta limiti critici per lo scaling:

Colli di bottiglia identificati:
  • Processing AI: Analisi bandi è CPU/GPU intensive. Un singolo bando può richiedere 30-120s di elaborazione
  • Database queries: PostgreSQL su Supabase con query complesse su migliaia di bandi rallenta oltre 10k record
  • Railway limits: Memory caps (512MB-8GB) e cold starts impattano performance
  • Make automation: 1000 operations/mese nel free tier, rate limits su webhook
  • Concurrent users: Express single-threaded, Lovable frontend non ottimizzato per caching

Architettura Attuale vs Target

Attuale: Monolite Express → Supabase → AI processing sincrono

Target 1000+ utenti: Microservizi asincroni + caching + queue system

Soluzioni di Scaling

1. Database Optimization (Immediate - Low Cost)

  • Indexing strategico: CREATE INDEX idx_bandi_deadline ON bandi(scadenza) WHERE stato='aperto';
  • Materialized views: Pre-calcolare statistiche comuni
  • Partitioning: Partizionare tabella bandi per regione/data
  • Connection pooling: PgBouncer per gestire connessioni
  • Read replicas: Supabase Pro offre replica read-only

Costo: €25/mese (Supabase Pro) | Gain: 3-5x query speed

2. Async Processing Queue (Critical)

Separare analisi AI dal request/response cycle:

  • BullMQ + Redis: Queue system per job AI processing
  • Worker pools: Scalare workers indipendentemente da API
  • Priority queue: Bandi urgenti (scadenza <7gg) in fast lane
// Esempio architettura
POST /bandi/analyze → Job in queue → 202 Accepted
Workers (3-10) → Process in parallel → Webhook callback
GET /bandi/:id/status → Real-time progress

Stack: Railway Redis + BullMQ | Costo: €10-30/mese

3. Caching Strategy (High Impact)

  • Redis cache: Risultati analisi bandi (TTL 24h)
  • CDN: Vercel Edge per frontend Lovable → Cloudflare
  • API caching: HTTP cache headers su endpoint read-only
  • Computed fields: Cache score compliance in DB

Hit rate target: 70%+ su query ripetute | Latency: 50ms vs 2000ms

4. AI Processing Optimization

  • Batch processing: Analizzare 10 bandi in parallelo invece di sequenziale
  • Model caching: Riutilizzare embeddings per bandi simili
  • Streaming responses: SSE per feedback real-time durante analisi
  • Fallback models: GPT-3.5 per analisi veloci, GPT-4 solo se necessario

Costo AI: Attuale $0.50/bando → Target $0.15/bando

5. Infrastructure Scaling (Railway → Hybrid)

Opzione A - Railway Pro:

  • Scalare a 2-4 instances Express
  • Load balancer automatico
  • Costo: €50-100/mese per 1000 utenti

Opzione B - Hybrid (Raccomandato):

  • Frontend: Lovable → Vercel (gratis fino 100GB bandwidth)
  • API: Railway Express (2 instances)
  • Workers AI: Modal Labs o Replicate (pay-per-use GPU)
  • DB: Supabase Pro con replicas

Costo totale: €80-150/mese | Capacity: 5000+ utenti

Monitoring & Observability

  • Sentry: Error tracking (free tier 5k events/mese)
  • Supabase Dashboard: DB performance metrics
  • Railway Metrics: CPU/Memory/Response times
  • Custom dashboards: Grafana + Prometheus per metriche business (bandi analizzati/ora, queue depth, cache hit rate)

Migration Path (4 settimane)

Week 1: DB optimization + indexing

Week 2: Implementare Redis + BullMQ queue

Week 3: Refactor AI processing → async workers

Week 4: Load testing + monitoring setup

Risorse Tecniche

Prossimi Step per Ambra

  1. Audit attuale: Analizzare Railway metrics ultimi 30gg - identificare peak usage patterns
  2. DB profiling: Eseguire EXPLAIN ANALYZE su query più lente, aggiungere indici
  3. Prototipo queue: Implementare BullMQ per 1 endpoint (/analyze) come POC

  6. Decision point: Railway scaling vs hybrid architecture - valutare con metriche reali
Red flags da monitorare:
  • Response time API >2s in p95
  • DB connection pool saturation (>80% used)
  • Railway memory >75% sustained
  • Make.com operations >800/mese (vicino al limite)
  • AI costs >30% revenue per utente
